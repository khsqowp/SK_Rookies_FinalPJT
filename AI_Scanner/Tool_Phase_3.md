# TOOL PHASE 3: Deep Logic & Auth Verification (심층 논리 검증)

---

## 0. 절대 규칙

**이 Phase에서 수행하는 것:**
- Phase 2 `candidates/` 의심 신호에 대한 심층 재현 및 확증
- 인증/인가 로직 교차 검증 (토큰 매트릭스)
- IDOR / BAC / Race Condition 확정 테스트
- 확정된 취약점만 `vulnerabilities/` 저장

**이 Phase에서 절대 수행하지 않는 것:**
- ❌ Phase 1, 2 재수행
- ❌ 통신 없이 뇌피셜로 "차단될 것 같다" 판단 후 스킵
- ❌ Phase 4(점수 산정) 진행

**뇌피셜(Hallucination) 절대 금지:**
- "스프링 시큐리티가 막았을 것", "프레임워크 차원에서 방어할 것" 등 실제 통신 없이 PASS 처리 금지.
- 403 Forbidden을 받더라도 **올바른 차단인지, 설정 오류인지** 반드시 다른 권한 토큰으로 추가 교차 검증.

**Reasoning Log 원칙:**
- 가설 먼저 → 검증 실시 순서 준수.
- 예: "STAFF 토큰으로 /api/admin/staff에 접근이 허용된다면 권한 상승이 가능하다" → `logs/reasoning.log` 기록 후 즉각 검증.

## 2. 토큰 매트릭스 구성 (사전 준비 필수)

> 이 Phase 시작 전 모든 권한 계층의 토큰을 확보하여 교차 검증 행렬을 구성한다.

**확보해야 할 토큰 계층:**
- ADMIN (최고 관리자)
- MANAGER (중간 관리자)
- STAFF (하위 직원)
- 일반 USER
- 비인증 (No Token)

**교차 검증 행렬 작성 방법:**

| Endpoint | ADMIN | MANAGER | STAFF | USER | NONE | 기대 결과 | 실제 결과 | 판정 |
|----------|-------|---------|-------|------|------|-----------|-----------|------|
| `POST /api/admin/staff` | 200 | 403 | **?** | 403 | 401 | STAFF → 403 | STAFF → 200 | ⚠️ 취약 |
| `GET /api/users/{타인ID}` | 200 | ? | ? | **?** | 401 | USER → 403 | USER → 200 | ⚠️ IDOR |

- **기대 결과와 실제 결과가 다를 경우** → 취약점 확정 후 `vulnerabilities/`에 저장
- **기대와 일치할 경우** → `mitigated/`에 방어 확인 기록

## 3. 취약점 유형별 분석 포인트

### A. BAC (Broken Access Control) [SK 4-5]

**분석 관점:**
- STAFF 계정으로 ADMIN 전용 기능(계정 생성, 권한 변경, 시스템 설정 등)을 실제로 수행할 수 있는가
- 단순히 응답 코드(403)만 확인하는 것이 아니라, **실제로 작업이 수행되었는지**(DB 변경, 계정 생성 등)를 확인해야 한다
- 수직 권한 상승(낮은 권한 → 높은 권한 기능 접근)과 수평 권한 상승(동일 권한 수준의 타인 자원 접근)을 모두 점검

**검증 완료 기준:**
- 낮은 권한 토큰으로 요청 전송 → HTTP 2xx 응답 → 실제 작업 수행 결과 확인 (예: 생성된 계정으로 로그인 가능)

---

### B. IDOR (Insecure Direct Object Reference) [SK 1-3]

**분석 관점:**
- URL 경로, Query String, Request Body에 있는 ID 파라미터를 타인의 ID로 치환했을 때 해당 자원이 반환되는가
- 조회(GET)뿐 아니라 **수정(PUT/PATCH), 삭제(DELETE)** 도 타인 자원에 대해 가능한가
- 자원의 종류별로 확인: 사용자 정보, 금융/거래 내역, 게시글, 파일, 주문 내역 등
- ID가 순차적(1, 2, 3...)인 경우, 예측 불가능한 UUID인 경우를 구분하여 공격 난이도 판단

**판단 기준:**
- 자신의 ID만 접근 가능하면 양호 / 타인의 ID로도 접근 가능하면 IDOR 취약점 확정

---

### C. Race Condition [SK 3-2]

**분석 관점:**
- 원자적으로 처리되어야 할 작업(잔액 차감, 재고 감소, 포인트 사용)에 동시 요청을 다수 발송했을 때 중복 처리가 발생하는가
- 요청 전 상태와 요청 후 상태를 비교하여 예상치보다 더 많은 자원이 소비/획득되었는가
- DB 트랜잭션 락(SELECT FOR UPDATE 등)이 적용되어 있는지 응답 패턴으로 추론

**판단 기준:**
- 동시 요청 N건 발송 후 자원 상태가 `초기값 - 단일처리값 × N`으로 변화하면 Race Condition 취약점 확정

---

### D. JWT 인증 토큰 우회 [SK 4-3]

**분석 관점:**
- JWT의 알고리즘을 `none`으로 변조하여 서명 없이 요청 시 서버가 수락하는가
- JWT payload의 `role`, `sub` 등 권한 관련 클레임을 조작하여 서명을 재생성하지 않아도 수락되는가
- 만료된(exp 과거) 토큰으로 API가 정상 접근되는가
- HS256 방식에서 공개 RSA 키를 비밀키로 사용한 서명 조작(알고리즘 혼동 공격)이 가능한가

**판단 기준:**
- 서명 검증 실패 시 401/403 반환이면 양호 / 변조된 토큰으로 정상 응답이면 취약점 확정

---

### E. 비인증 상태 관리자 API 접근 [SK 3-4]

**분석 관점:**
- 인증 토큰 없이 `/api/admin/`, `/api/internal/`, `/api/manage/` 등 관리자 경로에 직접 접근했을 때 200 응답이 반환되는가
- 응답 코드가 401(인증 필요)인지, 403(권한 부족)인지, 200(접근 허용)인지 구분
- 401이 아닌 403이 반환된다면, 인증 없이 경로 존재 자체는 확인되므로 추가 우회 시도 필요

## 4. 검증 완료 기록 규칙

**확정 취약점 저장:** `vulnerabilities/[SK항목번호]_[유형]/[ID]_Confirmed_[Name].md`

**파일 필수 포함 항목:**
- 취약점 ID, SK 항목, 유형, 심각도
- 원본 신호 파일 참조 경로
- 검증 절차 (몇 단계로 어떻게 확인했는가)
- 재현 가능 여부 및 재현 횟수
- 실제 요청/응답 증거 (HTTP 상태 코드 + 응답 본문 핵심 부분)
- 영향도 서술

**방어 확인 기록:** `mitigated/[SK항목번호]_[패턴].json`
- 403/401로 정상 차단된 항목은 Positive Finding으로 기록
