# TOOL PHASE 5: Advanced Reporting (최종 보고서 작성)

---

## 0. 절대 규칙

**이 Phase에서 수행하는 것:**
- Phase 1~4 결과 통합 → `FINAL_REPORT.md` 작성 (SK 28개 항목 통합 보고서)
- 취약점 개수만큼 독립 보고서 생성 → `reports/{ID}_{취약점명}.md`
- `logs/FINAL_INDEX.json` 생성

**이 Phase에서 절대 수행하지 않는 것:**
- ❌ Phase 1~4 재수행 또는 새로운 취약점 발견 시도
- ❌ Docx/PDF 등 포맷 변환 (Markdown 전용)
- ❌ 임의 목차 생성 (SK 28개 표준 항목 목차 준수)

**보고서 네이티브 원칙:**
- 이미지·스크린샷 대신 HTTP 요청/응답 원문을 코드블록으로 첨부.
- AI가 추정하여 작성한 응답 내용은 포함 금지. Phase 3 실증 기록만 사용.

## 2. 최종 산출물 구조 (2종)

```
[산출물 1]  FINAL_REPORT.md          ← SK 28개 항목 기반 통합 보고서 (1,000줄 이상)
[산출물 2]  reports/
            ├── BAC-001_Staff_to_Admin.md     ← 취약점 1건 독립 보고서
            ├── XSS-001_Stored_Community.md   ← 취약점 1건 독립 보고서
            ├── IDOR-001_User_AssetAccess.md  ← 취약점 1건 독립 보고서
            └── ...                           ← 확정 취약점 N개만큼 생성
```

> **취약점 독립 보고서는 하나의 취약점을 단독으로 클라이언트에게 전달할 수 있는 완전한 문서여야 한다.**
> 통합 보고서의 요약이 아니라, 해당 취약점의 모든 기술 정보를 독립적으로 담는다.

## 4. FINAL_REPORT.md: 각 취약점 섹션 작성 포인트

### 4.1 현황 및 문제점
- 어떤 기능/엔드포인트에서 취약점이 발생하는가
- 서버가 수행하지 않은 검증이 무엇인가 (인가 확인, 입력 검증, 인코딩 등)
- 취약한 코드 로직의 기술적 원인을 명확하게 서술

### 4.2 실제 검증 방법 (Verification Methodology)
- 어떤 권한 계정으로, 어떤 전략 순서로 확인했는가 Step-by-Step 서술
- 각 스텝에서 어떤 결과를 얻었고 왜 그것이 취약점의 증거인가
- **단순 행위 나열 금지. 각 스텝에 "왜"에 대한 분석 포함 필수**

### 4.3 공격 시나리오
- 실제 공격자가 이 취약점을 악용하는 단계적 흐름
- 피해 범위: 개인 데이터 노출 / 시스템 전체 장악 / 금전 손실 등 구체적 피해 서술

### 4.4 증명 (PoC)
- HTTP 요청/응답 원문을 코드블록으로 첨부 (AI 추정 값 사용 금지)
- 요청: Method, URL, Headers, Body
- 응답: HTTP Status, 핵심 응답 본문

### 4.5 근본 원인 (Root Cause)
- 코드·설정 레벨의 기술적 원인 명시
- 어노테이션 누락, 화이트리스트 미적용, 인코딩 누락 등

### 4.6 대응 방안 (Remediation)
- 코드/설정 수정 방향 (언어·프레임워크에 맞는 구체적 가이드)
- 패치 후 재테스트 방법 명시

## 5. 취약점 독립 보고서 (reports/{ID}_{Name}.md) 작성 포인트

> **FINAL_REPORT.md와 중복되어도 무관. 이 문서는 완전히 독립적으로 읽힐 수 있어야 한다.**

### 5.1 독립 보고서 필수 섹션

```
# {취약점 ID} - {취약점명}

## Executive Summary (1페이지 요약)
- 한 문단으로 이 취약점이 무엇이고, 왜 위험하며, 무엇을 해야 하는지 요약

## 취약점 개요
| 항목 | 내용 |
|------|------|
| ID | BAC-001 |
| 위험도 | CRITICAL |
| CVSS v3.1 | 9.9 (AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H) |
| SK 항목 | 4-5 |
| CWE | CWE-285 |
| OWASP | A01:2021 Broken Access Control |
| 발견 엔드포인트 | POST /api/admin/staff |
| 영향 범위 | 전체 시스템 관리자 권한 탈취 가능 |

## 기술 분석
### 취약한 로직 흐름
[취약점이 발생하는 서버 내부 처리 흐름을 단계별로 설명]

### 취약 원인 코드 (추정)
[코드 레벨 원인 서술 - 실제 소스 접근 불가 시 블랙박스 관점에서 추정]

## 상세 재현 절차 (Step-by-Step PoC)
### Step 1: {설명}
[요청/응답 코드블록]

### Step 2: {설명}
[요청/응답 코드블록]

### Step 3: 피해 확인
[최종 피해 증거 코드블록]

## 비즈니스 영향 분석
### 직접 영향
[데이터 유출, 거래 조작, 서비스 중단 등 즉각적 피해]

### 간접 영향
[규제 위반, 평판 손실, 법적 책임 등 간접 피해]

### 위험 시나리오
[최악의 경우 (Worst-Case Scenario) 서술]

## 수정 가이드 (Remediation Guide)
### 즉시 조치 (임시 방편)
[배포 전 즉시 적용 가능한 임시 차단 방법]

### 근본적 수정
[코드 레벨 수정 방향 및 패턴]

### 수정 후 검증 방법
[어떻게 재테스트해야 수정이 완료됨을 확인할 수 있는가]

## 유사 취약점 추가 점검 권고
[이 취약점과 동일한 패턴으로 발생할 수 있는 다른 엔드포인트/기능 목록]
```

### 5.2 독립 보고서 생성 조건

- `vulnerabilities/` 폴더에 저장된 **확정 취약점 1건 = 독립 보고서 1개**
- `triage_summary.json`의 `detailed_findings` 배열 순회하여 자동 생성
- 파일명: `reports/{ID}_{취약점명_영문축약}.md`
  - 예: `reports/BAC-001_Staff_AdminCreate.md`
  - 예: `reports/XSS-001_Stored_Community.md`
  - 예: `reports/IDOR-001_User_AssetAccess.md`

## 6. 보고서 생성 절차

```
Step 1: triage_summary.json 읽기
        → detailed_findings[] 배열 기준으로 확정 취약점 목록 파악

Step 2: FINAL_REPORT.md 생성
        → SK 28개 항목 목차 구조로 통합 보고서 작성
        → 3.3절에 독립 보고서 파일 목록 링크 삽입

Step 3: 각 확정 취약점별 독립 보고서 생성
        → reports/{ID}_{Name}.md × N개 (N = 확정 취약점 수)
        → 각 보고서는 Section 5.1의 독립 보고서 섹션 구조로 작성

Step 4: logs/FINAL_INDEX.json 생성
        → 생성된 모든 산출물 목록 기록
```

**FINAL_INDEX.json 포맷:**
```json
{
  "report_date": "",
  "total_endpoints_found": 0,
  "total_attack_attempts": 0,
  "confirmed_vulnerabilities": 0,
  "false_positives": 0,
  "mitigated_count": 0,
  "severity_breakdown": { "CRITICAL": 0, "HIGH": 0, "MEDIUM": 0, "LOW": 0 },
  "output_files": {
    "consolidated_report": "FINAL_REPORT.md",
    "individual_reports": [
      "reports/BAC-001_Staff_AdminCreate.md",
      "reports/XSS-001_Stored_Community.md"
    ]
  }
}
```
