# TOOL PHASE 2: Advanced Fuzzing & Scanning (전수 공격 벡터 스캔)

---

## 0. 절대 규칙

**이 Phase에서 수행하는 것:**
- Phase 1에서 발견된 모든 Endpoint × 전체 공격 벡터 전수 테스트
- 의심 신호(Suspect) 파일 생성
- 모든 요청/응답 로그 저장

**이 Phase에서 절대 수행하지 않는 것:**
- ❌ Phase 3(Logic 검증) 진행 또는 취약점 확정 판정
- ❌ 반응 없음을 추측으로 판단 후 스킵 (반드시 5 탐침 전송 후 결과로 판단)

**Anti-Laziness 원칙:**
- 각 URL × 각 벡터를 개별로 수행. 묶어서 처리 금지.
- 탐침 선택 사유를 `logs/reasoning.log`에 선행 기록.

**오탐(FP) > 미탐(FN) 원칙:**
- 반응이 미약하더라도 의심 신호 파일 생성. 신호 수집이 목적.

## 1. 진단 항목 매핑 (SK 標準)

| SK 항목 | 공격 벡터 | 핵심 분석 관점 |
|---------|----------|--------------|
| **1-1** | XSS / CSRF | 입력값이 HTML로 반사·저장되는가 / CSRF 토큰 없이 상태 변경 요청 가능한가 |
| **1-2** | SQL Injection | DB 쿼리에 구문 주입하여 응답이 변화하는가 |
| **1-3** | 파라미터 조작 / Mass Assignment | 파라미터 조작·추가 시 서버 동작이 변화하는가 |
| **1-4** | XXE (XML External Entity) | XML 입력값에 외부 Entity를 삽입하여 서버 내부 리소스를 읽을 수 있는가 |
| **1-5** | SSRF (Server-Side Request Forgery) | 서버가 공격자 지정 URL에 요청을 보내는가 |
| **1-6** | 비즈니스 로직 우회 | 음수·한계치·동시 요청으로 정상 흐름을 이탈할 수 있는가 |
| **2-1** | 악성 파일 업로드 | 허용되지 않은 확장자·MIME 위장 파일을 업로드·실행할 수 있는가 |
| **4-2** | Open Redirect | 리다이렉트 파라미터 조작으로 외부 악성 도메인으로 유도 가능한가 |
| **4-4** | Command Injection | 입력값을 통해 OS 명령이 실행되는가 |
| **4-6** | Path Traversal | 상위 디렉터리 이탈로 의도치 않은 파일에 접근할 수 있는가 |

## 2. 핵심 전략: 5 탐침 → 반응 분석 → 심층 탐색

### 전체 흐름 (의사코드)

```
FOR EACH endpoint IN Phase 1 발견 Endpoint 목록:
  FOR EACH vector IN 전체 공격 벡터 목록:

    [1단계] logs/reasoning.log에 이 파라미터 × 이 벡터를 선정한 사유 선행 기록

    [2단계] 스크립트 실행으로 5 탐침 발송
      → ./scripts/probe_fuzz.sh {endpoint} {vector} {param}
      → 결과: 각 탐침별 (상태코드, 응답길이, 오류메시지, 응답시간) 요약만 반환

    [3단계] 반응 분석 (AI 판단)
      IF 반응 감지:
        → [4단계] 심층 탐색 모드 진입 (AI가 응답 패턴 기반 적응형 추가 시도)
        → Suspect 파일 생성
      ELSE:
        → 다음 벡터로 이동

  END FOR
END FOR
```

## 3. 스크립트 역할 정의

### `./scripts/probe_fuzz.sh` (탐침 전용 스크립트)

**역할:** 각 벡터의 카테고리별 탐침 1개씩 발송 후 응답을 요약만 반환. 원본 HTML/Body는 AI에게 전달하지 않아 토큰 절감.

**반환 형식:**
```
[probe_1] 상태:200 길이:1234 오류메시지:없음  응답시간:0.12s
[probe_2] 상태:200 길이:1240 오류메시지:없음  응답시간:0.11s
[probe_3] 상태:500 길이:892  오류메시지:"SQL syntax error near..." 응답시간:0.14s  ← 반응!
[probe_4] 상태:200 길이:1234 오류메시지:없음  응답시간:5.03s  ← 시간 지연 반응!
[probe_5] 상태:200 길이:2100 오류메시지:없음  응답시간:0.13s  ← 길이 이상!
```

## 4. 벡터별 5 탐침 카테고리 정의

### 벡터 1: XSS (Cross-Site Scripting) [SK 1-1]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | Script 태그 기반 | `<script>` 패턴이 그대로 반영되는가 |
| probe_2 | 이벤트 핸들러 기반 | `onerror`, `onload` 등이 필터링되는가 |
| probe_3 | SVG / HTML5 기반 | SVG 태그를 통한 우회 가능성 |
| probe_4 | 인코딩 기반 | URL 인코딩 시 필터 우회 가능한가 |
| probe_5 | 속성값 탈출 기반 | 속성 내부에서 따옴표 탈출 가능한가 |

**반응 기준:** 응답에 페이로드 패턴이 HTML 인코딩 없이 그대로 반영되는가

**심층 탐색 추가 분석:**
- Stored XSS: 저장 후 다른 사용자가 조회하는 시나리오까지 확인
- DOM-based XSS: URL Fragment(`#`) 또는 `location.hash` 조작 시 JS 실행 여부
- CSP가 설정된 경우: CSP 정책의 허용 도메인에서 스크립트 로드 가능한가 (`unsafe-inline`, `unsafe-eval` 여부)

---

### 벡터 2: CSRF (Cross-Site Request Forgery) [SK 1-1]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | CSRF 토큰 존재 여부 | 상태 변경 요청(POST/PUT/DELETE)에 CSRF 토큰이 요구되는가 |
| probe_2 | CSRF 토큰 누락 요청 | CSRF 토큰 없이 상태 변경 요청 전송 시 거부되는가 |
| probe_3 | CSRF 토큰 재사용 | 이미 사용한 CSRF 토큰을 재사용할 수 있는가 |
| probe_4 | Referer 우회 | Referer 헤더를 타 도메인으로 변조 시 요청이 수락되는가 |
| probe_5 | SameSite 쿠키 속성 | 세션 쿠키에 `SameSite=Strict/Lax` 속성이 없어 CSRF 가능한가 |

**반응 기준:** CSRF 토큰 없이 상태 변경 요청 성공 시 취약점 신호

---

### 벡터 3: SQL Injection [SK 1-2]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | Error-based | SQL 문법 오류 메시지가 응답에 노출되는가 |
| probe_2 | Boolean-based | 논리 조건 변경 시 응답 내용이 달라지는가 |
| probe_3 | Time-based Blind | Sleep 함수 삽입 시 응답 시간이 유의미하게 지연되는가 |
| probe_4 | UNION-based | UNION SELECT 삽입 시 추가 데이터가 반환되는가 |
| probe_5 | 인코딩 우회 | URL·주석·공백 인코딩으로 필터 우회 가능한가 |

**반응 기준:** SQL 오류 메시지 노출 / 응답 내용 변화 / 3초 이상 응답 지연

**심층 탐색 추가 분석:**
- Second-Order SQLi: 입력 시에는 정상이지만 나중에 조회/사용 시 SQL 실행되는 경우
- NoSQL Injection: MongoDB 등 NoSQL 환경에서 `$where`, `$ne` 연산자를 이용한 주입

---

### 벡터 4: 파라미터 조작 / Mass Assignment [SK 1-3]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 숫자 극단값 | 음수, 0, 최대 정수 삽입 시 처리하는가 |
| probe_2 | 불린 반전 | `is_admin`, `enabled` 등 불린 파라미터 반전 시 동작 변화 |
| probe_3 | 권한 문자 조작 | `role`, `status` 파라미터에 상위 권한값 삽입 |
| probe_4 | Mass Assignment | 요청 Body에 `is_admin`, `role`, `verified` 등 미노출 파라미터 추가 삽입 |
| probe_5 | 타입 혼동 | 숫자 파라미터에 문자열·배열·객체 등 타입 불일치 값 삽입 |

**반응 기준:** 응답 코드 변화 / 로직 동작 변화 / 계정 권한 변화

**심층 탐색 추가 분석:**
- Mass Assignment 심층: 응답(GET) 에 포함된 모든 필드를 PUT/PATCH Body에 그대로 전송했을 때 서버가 수정 불가 필드를 업데이트하는가 (`readonly` 필드 우회)

---

### 벡터 5: XXE (XML External Entity) [SK 1-4]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 기본 External Entity | `<!ENTITY xxe SYSTEM "file:///etc/passwd">` 삽입 후 파일 내용 반환 여부 |
| probe_2 | Parameter Entity | DTD를 통한 간접 Entity 선언 방식 우회 |
| probe_3 | OOB (Out-of-Band) | 응답에 직접 반영 안 되더라도 외부 서버로 데이터 전송 여부 (Blind XXE) |
| probe_4 | SSRF via XXE | XXE를 통해 내부 서비스(`http://169.254.169.254/` 등)에 요청 유도 |
| probe_5 | MIME 우회 | JSON 요청이지만 Content-Type을 XML로 변경 시 서버가 XML로 파싱하는가 |

**반응 기준:** 파일 내용 응답 포함 / Blind 응답시간 지연 / 외부 DNS 쿼리 발생

---

### 벡터 6: SSRF (Server-Side Request Forgery) [SK 1-5]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 로컬호스트 접근 | URL 파라미터로 `http://localhost` 또는 `http://127.0.0.1` 요청 유도 가능한가 |
| probe_2 | 내부망 접근 | 내부 IP 대역(`http://192.168.x.x`, `http://10.x.x.x`) 접근 가능한가 |
| probe_3 | 클라우드 메타데이터 | `http://169.254.169.254/latest/meta-data/` (AWS IMDSv1) 접근 가능한가 |
| probe_4 | 프로토콜 우회 | `file://`, `dict://`, `gopher://` 등 비HTTP 프로토콜 사용 가능한가 |
| probe_5 | DNS Rebinding | 외부 DNS를 통해 내부 IP로 해석되는 도메인으로 우회 가능한가 |

**반응 기준:** 내부 주소 응답 반환 / 응답 지연 차이 / Blind SSRF는 외부 Collaborator 응답 수신

**심층 탐색 추가 분석:**
- Webhook, Redirect, Fetch, Proxy, Import 기능에서 URL 파라미터를 받는 경우 우선 테스트

---

### 벡터 7: 비즈니스 로직 우회 [SK 1-6]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 음수 금액 | 결제/이체 금액에 음수 삽입 시 잔액이 역방향으로 증가하는가 |
| probe_2 | 한계치 초과 | 잔액보다 큰 금액 출금 요청 시 서버가 차단하는가 |
| probe_3 | 쿠폰·할인 중복 사용 | 동일 쿠폰을 반복 사용하거나 여러 할인을 중복 적용할 수 있는가 |
| probe_4 | 단계 건너뛰기 | 순서 의존 워크플로우를 중간 단계 없이 직접 호출 가능한가 |
| probe_5 | Race Condition | 동일 요청 다수 병렬 발송 시 원자성 위반으로 중복 처리가 발생하는가 |

**반응 기준:** 데이터 상태 변화 (잔액·재고·포인트가 예상 외로 변동)

---

### 벡터 8: 악성 파일 업로드 [SK 2-1]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 확장자 블랙리스트 우회 | 실행 파일 확장자 그대로 업로드 허용되는가 |
| probe_2 | MIME 타입 위장 | Content-Type을 이미지로 위장 후 실행 파일 업로드 |
| probe_3 | 이중 확장자 | `.php.jpg`, `.jsp.png` 형태로 확장자 검사 우회 가능한가 |
| probe_4 | Null Byte / 특수문자 | 확장자 뒤 Null Byte 삽입으로 우회 가능한가 |
| probe_5 | 파일 실행 확인 | 업로드 성공 후 저장 경로로 직접 접근 시 코드가 실행되는가 |

**추가 심층 분석:**
- SVG 파일을 통한 Stored XSS (SVG 내부 `<script>` 태그)
- 업로드된 파일 경로가 예측 가능한가 (IDOR와 연계)

### 벡터 9: Open Redirect [SK 4-2]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 직접 URL 삽입 | `redirect=https://evil.com` 파라미터로 외부 도메인 리다이렉트 가능한가 |
| probe_2 | 상대 경로 우회 | `//evil.com` (프로토콜 생략) 형태로 우회 가능한가 |
| probe_3 | URL 인코딩 우회 | `%68%74%74%70%73` 등 인코딩으로 필터 우회 가능한가 |
| probe_4 | 도메인 혼동 | `https://legit.com.evil.com` 형태로 도메인 필터 우회 가능한가 |
| probe_5 | 피싱 연계 | 리다이렉트 성공 시 정상 도메인 URL로 보이는 피싱 링크 생성 가능한가 |

### 벡터 10: Command Injection [SK 4-4]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 세미콜론 구분자 | 명령 뒤 세미콜론으로 추가 명령 실행 여부 |
| probe_2 | 파이프 구분자 | `\|` 연산자로 명령 연결 실행 여부 |
| probe_3 | 서브셸 | `$()` 또는 백틱으로 명령 치환 실행 여부 |
| probe_4 | AND/OR 구분자 | `&&`, `\|\|` 논리 연산자로 조건부 실행 여부 |
| probe_5 | 인코딩 우회 | URL 인코딩된 구분자로 필터 우회 가능한가 |

### 벡터 11: Path Traversal [SK 4-6]

| 탐침 # | 카테고리 | 확인하는 것 |
|--------|---------|------------|
| probe_1 | 기본 상위 이동 | `../` 패턴 삽입 시 상위 경로 접근 가능한가 |
| probe_2 | URL 인코딩 | `%2e%2e%2f` 형태로 필터 우회 가능한가 |
| probe_3 | 이중 인코딩 | `%252e%252e%252f` 이중 인코딩으로 우회 가능한가 |
| probe_4 | Null Byte | Null Byte 삽입으로 확장자 검사 우회 가능한가 |
| probe_5 | 절대 경로 직접 입력 | 절대 경로 삽입 시 대상 파일이 반환되는가 |

## 5. 반응 기준 (스크립트 반환값 해석)

| 신호 | 조건 | 의미 |
|------|------|------|
| 상태코드 변화 | 정상 200 → 500/403/400 | 서버 로직이 영향받음 |
| 응답 길이 변화 | ±20% 이상 차이 | 반환 데이터가 변화함 |
| 응답 시간 지연 | 3초 이상 | Time-based 공격 가능성 |
| 오류 메시지 노출 | SQL/파일 경로/스택 트레이스 포함 | 내부 정보 노출 |
| 페이로드 반영 | 응답에 입력 패턴 그대로 포함 | XSS 가능성 |
| 외부 연결 | 외부 DNS/HTTP 요청 발생 | Blind SSRF/XXE 가능성 |

## 6. 심층 탐색 모드 (Deep Exploration)

> 5 탐침 중 1개 이상 반응 감지 시 즉시 심층 탐색 모드로 진입.

**심층 탐색 원칙:**
- AI가 응답 패턴을 직접 분석하여 다음 시도 방향을 결정
- 어떤 탐침에서 반응이 나왔는지, 어떤 부분이 필터링되었는지 역추론
- 필터 우회 방향을 좁혀가며 최대 10회 추가 시도
- 우회 성공 또는 10회 소진 후에도 실패 시 "높은 가능성" 신호 파일 생성

---

## 7. 의심 신호(Suspect) 파일 생성 규칙

**파일 경로:** `candidates/[SK항목번호]_[유형]/[번호]_[신호명].md`

**파일 필수 포함 항목:**
- 반응 감지된 탐침 번호 및 카테고리
- 스크립트 반환 요약 (상태코드, 응답길이, 오류메시지, 응답시간)
- 심층 탐색 시 추가 시도 내용 및 각 결과 요약
- 위험도 추정 및 근거
- 해당 SK 항목 번호

## 8. 검증 (Self-Validation)

- [ ] 모든 Endpoint × 11 벡터에서 5 탐침 전송이 완료되었는가?
- [ ] 탐침은 `probe_fuzz.sh` 스크립트를 통해 발송하고 요약 결과만 수신했는가?
- [ ] 반응 감지 시 AI 주도 심층 탐색 모드로 전환했는가?
- [ ] CSRF, XXE, SSRF, Open Redirect 벡터가 포함되었는가?
- [ ] 비즈니스 로직(쿠폰 중복, Race Condition 등, SK 1-6) 탐침을 수행했는가?
- [ ] 파일 업로드의 실행 확인(저장 경로 접근)까지 수행했는가?
- [ ] Mass Assignment (미노출 파라미터 삽입) 탐침을 수행했는가?
- [ ] 탐침 선택 사유가 `logs/reasoning.log`에 선행 기록되었는가?
- [ ] 신호 파일에 탐침 결과 요약과 심층 탐색 내용이 포함되었는가?
