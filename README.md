# 프로젝트 소개 및 보안 조치 개요 보고서 — VCE

- **프로젝트명**: VCE (Virtual Currency Exchange)
- **작성일**: 2026-04-13
- **서비스 구성**: Exchange / Bank / Admin
- **대상 기술 스택**: React (Vite) + Spring Boot + Oracle DB + Docker

---

## 개요

VCE는 가상자산 거래소, 은행, 관리자 백오피스를 하나의 공통 백엔드 위에 구성한 통합형 웹 프로젝트입니다. 핵심 목적은 거래 기능이 있는 서비스를 구현하는 것에 더해, 실제 운영 과정에서 발생할 수 있는 보안 취약점을 식별하고 이를 어떻게 줄일지 코드와 문서 수준에서 함께 정리하는 데 있습니다.

서비스는 세 영역으로 구성됩니다. 거래소는 시세 조회, 자산 조회, 주문, 거래내역, 커뮤니티, 이벤트, 고객지원 기능을 제공하고, 은행은 계좌 조회, 입금, 출금, 이체 같은 금융 기능을 담당합니다. 관리자 서비스는 회원 관리, 거래 조회, 신분증 승인, 자산 회수, 문의 처리, 직원 관리처럼 운영상 민감한 기능을 전담합니다. 이 세 서비스는 하나의 화면 안에서 단순 분기로 처리하지 않고, 프론트엔드 모드와 서비스 경로를 분리해 설계했습니다.

VCE의 중요한 특징은 기능 구현과 보안 조치를 함께 다룬다는 점입니다. 입력값 검증, 인증 및 인가 강화, 관리자 기능 분리, 민감정보 마스킹, 파일 업로드 통제, 보안 헤더 설정 같은 조치를 실제 수정 이력과 함께 정리해 두었기 때문에, 단순 구현 결과물이 아니라 "어떤 위험을 어떤 방식으로 줄였는가"까지 추적할 수 있는 프로젝트입니다.

## 1. 프로젝트 목표

| 항목 | 내용 |
|------|------|
| 핵심 목표 | 가상자산 서비스 구현 + 취약점 진단 및 조치 이행 |
| 구현 범위 | 거래소, 은행, 관리자 백오피스 |
| 보안 관점 목표 | 인증/인가, 입력 검증, 파일 처리, 민감정보 노출 통제, 관리자 권한 분리 |
| 결과물 형태 | 서비스 코드 + 보안 조치 보고서 + 아키텍처/명세 문서 |

### 주요 지향점

1. **기능 구현**: 실제 서비스처럼 보이는 흐름을 가진 거래소/은행/관리자 기능 구현
2. **보안 진단**: 웹 취약점 진단 항목을 기준으로 서비스 구조 점검
3. **조치 이행**: 발견된 취약 가능성에 대해 코드와 설정 수준의 개선 적용
4. **문서화**: 조치 내용을 보고서 형식으로 정리해 재검토 가능한 형태로 유지

## 2. 서비스 구성

### Exchange

| 항목 | 내용 |
|------|------|
| 대상 사용자 | 일반 사용자 |
| 서비스 성격 | 가상자산 거래 메인 서비스 |
| 주요 기능 | 시세 조회, 자산 조회, 주문, 거래내역, 커뮤니티, 이벤트, 고객지원 |

거래소 서비스는 사용자 접점이 가장 넓은 영역입니다. 사용자는 시세를 확인하고 자산 현황을 조회하며, 주문을 생성하거나 취소할 수 있습니다. 이와 함께 커뮤니티, 이벤트, FAQ, 1:1 문의 기능까지 포함해 단순한 거래 화면이 아니라 플랫폼형 서비스 구조를 구성했습니다.

### Bank

| 항목 | 내용 |
|------|------|
| 대상 사용자 | 일반 사용자 |
| 서비스 성격 | 금융 기능 분리 서비스 |
| 주요 기능 | 계좌 조회, 입금, 출금, 이체 |

은행 서비스는 거래소와 연결되지만 더 민감한 금융 기능을 별도로 분리한 영역입니다. 계좌 조회, 입출금, 계좌 이체와 같은 흐름을 담당하며, 계좌 소유 여부를 서버에서 다시 검증해 단순한 화면 접근만으로는 금융 데이터나 타인 계좌를 조작할 수 없도록 구성했습니다.

### Admin

| 항목 | 내용 |
|------|------|
| 대상 사용자 | 운영자, 관리자, 직원 |
| 서비스 성격 | 관리자 전용 백오피스 |
| 주요 기능 | 회원 관리, 신분증 승인, 자산 회수, 문의 처리, 직원 관리 |

관리자 서비스는 일반 사용자 서비스 내부에 숨겨진 부가 메뉴가 아니라 완전히 분리된 운영 시스템입니다. 회원 상태 변경, 신분증 승인, 자산 회수, 직원 생성/삭제, 민감정보 조회 같은 고위험 기능이 포함되어 있기 때문에, 역할 분리와 접근 통제가 가장 엄격하게 적용되는 영역입니다.

## 3. 서비스 구조 및 설계 방향

| 항목 | 내용 |
|------|------|
| 프론트엔드 구조 | 하나의 코드베이스, 세 가지 앱 모드(`exchange`, `bank`, `admin`) |
| 백엔드 구조 | 하나의 Spring Boot REST API 서버 |
| 데이터 저장소 | Oracle DB |
| 파일 저장 방식 | 업로드 볼륨 경로 분리 |
| 배포 구조 | Docker / Compose 기반 서비스 단위 분리 |

이 프로젝트는 프론트엔드를 세 개로 논리 분리하면서도 백엔드를 공통으로 사용하는 구조를 취합니다. 이 방식은 중복 구현을 줄이면서도 서비스 목적별 경계를 유지하는 데 유리합니다. 사용자는 자신의 역할에 맞는 프론트엔드 서비스에 접속하고, 각 프론트엔드는 동일한 백엔드 API를 호출하지만 응답 범위와 접근 권한은 역할에 따라 달라집니다.

운영 관점에서는 프론트엔드, 백엔드, 데이터베이스를 컨테이너 단위로 분리해 구성함으로써 서비스별 배포 단위와 네트워크 경계를 나눌 수 있도록 했습니다. 특히 관리자 서비스는 사용자 서비스와 별도 포트 및 별도 프론트엔드 모드로 구성해, 단순 UI 숨김이 아니라 운영 경로 자체를 분리하는 방향으로 설계했습니다.

#### 기술 스택 선택 이유

프론트엔드는 사용자 화면과 관리자 화면 모두에서 상태 변화가 많고, 서비스 모드에 따라 다른 흐름을 제공해야 하기 때문에 컴포넌트 재사용성과 라우팅 유연성이 중요했습니다. React와 TypeScript 조합은 서비스 영역별 화면을 나누면서도 공통 컴포넌트를 유지하기에 적합했고, Vite는 `exchange`, `bank`, `admin` 같은 모드 분리 빌드를 처리하기에 효율적이었습니다. 또한 axios와 React Router DOM을 통해 인증 상태와 접근 경로를 일관되게 다룰 수 있도록 구성했습니다.

백엔드는 인증, 인가, 자산 처리, 주문 처리, 관리자 기능 등 비즈니스 로직이 집중되는 계층입니다. Spring Boot는 이러한 기능을 컨트롤러, 서비스, 리포지토리, 보안 설정 단위로 명확하게 나눌 수 있어 구조적 관리에 유리했습니다. 특히 Spring Security와 Validation, JPA 조합은 취약점 진단 프로젝트에서 중요한 인증/인가, 입력 검증, 데이터 무결성 확보를 체계적으로 반영하는 데 적합했습니다. OAuth2 Client와 JJWT를 함께 사용해 일반 로그인과 소셜 로그인 흐름을 동시에 지원할 수 있도록 구성한 점도 특징입니다.

데이터베이스는 프로젝트 요구사항과 실습 환경을 고려해 Oracle 계열을 사용했습니다. 거래와 자산, 회원, 문의 같은 데이터는 서비스 전반의 핵심 자산이기 때문에 안정적인 영속 저장소가 필요했고, JPA와 Oracle 조합으로 이를 관리했습니다. 인프라 측면에서는 Docker와 Compose를 사용해 프론트엔드, 백엔드, 데이터베이스를 분리된 단위로 실행함으로써 환경 재현성을 높이고, 서비스 구조 자체를 더 명확하게 유지할 수 있도록 했습니다. 프론트엔드 서빙에는 OpenResty 기반 구성을 사용해 정적 파일 배포와 SPA 라우팅을 안정적으로 처리하도록 했습니다.

## 5. 보안 조치 중심 개발

| 항목 | 내용 |
|------|------|
| 인증 방식 | JWT 기반 인증 |
| 인가 방식 | 역할 기반 접근 제어 |
| 세션 관리 | 관리자/사용자 세션 분리 |
| 입력 통제 | DTO 바인딩 + 유효성 검증 |
| 파일 통제 | 확장자 화이트리스트 + Magic Byte 검증 |
| 민감정보 보호 | 응답 마스킹 + 언마스킹 권한 분리 |
| 배포 보안 | 프로덕션 빌드 기반 정적 서빙 |

프로젝트에는 취약점 점검 항목을 실제로 반영한 조치가 포함되어 있습니다.

- JWT 기반 인증과 역할 기반 인가
- 관리자/사용자 세션 분리
- 입력값 검증과 DTO 바인딩
- 업로드 파일 확장자 및 Magic Byte 검증
- 민감정보 마스킹
- 관리자 전용 언마스킹 API
- 보안 헤더 적용
- 프로덕션 빌드 기반 정적 서빙

이 조치들은 기능 개발 이후에 임시로 붙인 보완책이 아니라, 서비스 구조를 정리하는 과정에서 함께 반영된 설계 요소입니다. 특히 관리자 서비스 분리, 세션 분리, 민감정보 응답 통제는 단순한 화면 처리 수준이 아니라 역할과 책임을 분리하는 구조적 조치라는 점에서 의미가 있습니다.

| 토큰/세션 | 관리자/사용자/은행 토큰 분리, 이전 토큰 무효화 정책 개선, 중복 로그인 조치, 멀티 도메인 로그인 세션 분리 정리 |
| 계정/비밀번호 | 기존 취약 계정 정리, BCrypt 기반 비밀번호 정책 강화, 회원가입 입력 형식 검증 수정, 회원가입 완료 토큰 흐름 무상태화 |
| 관리자 분리 | 관리자 페이지 별도 경로/별도 컨테이너 분리, 사용자 토큰과 관리자 토큰 처리 정리 |
| 저장소/세션 | 관리자 로컬 스토리지 정리, 세션 키 구조 정리, 관리자 유휴 로그아웃 및 세션 유지 정책 조정 |
| 문의/개인정보 | 문의 게시판 응답을 DTO 기반으로 변경해 주요 정보 노출 완화 |
| 입력 검증 | CommunityService JPQL 파라미터화, 관리자 검색/요청 값 검증 강화, 음수 금액 입력 검증, 이체 파라미터 검증 |
| 동시성/무결성 | race condition 조치, 트랜잭션 재시도 보강, 금융 처리 무결성 강화 |
| 업로드/XSS | 파일 업로드 검증 강화, 커뮤니티 XSS 취약점 조치 |
| 오류/노출 | 오류 메시지 노출 완화, 파일 경로 노출 차단, 서버/프레임워크/헤더 정보 노출 최소화 |
| 배포/검색 노출 | 컨테이너 분리, robots.txt 추가, 코드 직접 노출 제거 |

### 확인된 대표 수정 포인트

1. **문의 게시판 주요 정보 노출 취약점 수정**  
   문의 조회 응답을 엔티티 직접 반환이 아니라 DTO 기반 응답으로 바꾸어 불필요한 정보 노출을 줄였습니다.

2. **관리자 페이지 세션 및 로그인 흐름 정리**  
   관리자 로그인 실패 횟수 제한, 매니저/스태프 로그인 실패 버그 수정, 관리자 로컬 스토리지 정리, 유휴 로그아웃 및 세션 유지 정책 조정이 확인되었습니다.

3. **인증 분리 및 토큰 정책 강화**  
   거래소/은행 로그인 완전 분리, 관리자 토큰 비활성화 정리, 이전 토큰 무효화 정책 개선, 중복 로그인 조치, 멀티 도메인 세션 분리 정리가 반영되었습니다.

4. **입력값 및 인젝션 대응 강화**  
   바인딩 오류 수정, 회원가입 입력 형식 보정, CommunityService JPQL 파라미터화, 관리자 검색 요청 검증 강화, 음수 금액 검증, 이체 파라미터 검증이 반영되었습니다.

5. **무결성 및 동시성 보완**  
   race condition 조치와 트랜잭션 재시도 보강을 통해 금융 처리 과정의 무결성을 강화한 이력이 확인되었습니다.

6. **배포 구조와 노출 범위 정리**  
   컨테이너 분리, robots.txt 추가, Vite 개발 서버 기반 코드 노출 제거, 서버/프레임워크/헤더 정보 노출 차단이 확인되었습니다.

7. **업로드 및 콘텐츠 보안 강화**  
   파일 업로드 취약점 조치와 커뮤니티 XSS 취약점 조치가 코드리뷰 이력으로 확인되었습니다.

8. **회원가입 흐름 개선**  
   회원가입 입력 오류 수정, 만료 세션 처리 보완, 회원가입 완료 코드를 무상태 토큰 방식으로 변경한 이력이 확인되었습니다.

## 7. 서비스별 보안 관점 요약

### Exchange

- 커뮤니티 기능을 포함하기 때문에 XSS와 파일 업로드 취약점 대응이 중요
- 외부 인증 제공자 연동으로 OAuth2 리다이렉트 검증이 중요
- 자산 조회와 주문 API는 일반 사용자 인증 흐름의 안정성이 중요

### Bank

- 계좌 조회, 입출금, 이체 기능이 있어 금융 데이터 접근 제어가 중요
- 요청자가 실제 계좌 소유자인지 서버에서 재검증하는 구조가 핵심
- 단순 화면 분리만으로는 부족하므로 서버 측 권한 검증이 중요

### Admin

- 회원 관리, 자산 회수, 신분증 열람처럼 고위험 기능이 집중된 영역
- 관리자 역할 분리와 언마스킹 권한 통제가 핵심
- 관리자 전용 인증 흐름과 사용자 세션 분리가 필수
